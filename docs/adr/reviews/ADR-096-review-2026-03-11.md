## Kontext

ADR-096 formalisiert den Scope von `authoringfw` als Content Workflow Orchestration Package mit drei Domains (Writing, Research, Analysis). Es löst OQ-3 aus ADR-095 und baut auf dem `BaseContentOrchestrator`-Pattern aus ADR-095 §5.8 auf.

---

## [SUGGEST] S1 — `researchfw` vs `authoringfw.research` Abgrenzung

**Datei:** ADR-096:147-151
**Begründung:** Es existiert bereits `researchfw` als eigenständiges PyPI-Package (ADR-104/105). ADR-096 definiert `authoringfw/research/` als Sub-Modul für Research-Orchestrierung. Die Abgrenzung sollte explizit dokumentiert werden:
- `researchfw` = Low-Level Research-Infrastruktur (API-Clients, Source-Management, Rate-Limiting)
- `authoringfw.research` = High-Level Orchestrierung (LLM-basierte Synthese + Analyse via `aifw`)

Empfehlung: Ein kurzer §4.9 "Relationship to `researchfw` (ADR-104)" analog zu §4.8 (ADR-068).

---

## [SUGGEST] S2 — `ContentTask.to_context()` → Protocol statt ABC

**Datei:** ADR-096:266-272
**Begründung:** `ContentTask` als Pydantic `BaseModel` mit `@abstractmethod` funktioniert, aber ein `Protocol` wäre idiomatischer für Pydantic v2:
```python
class HasContext(Protocol):
    def to_context(self) -> dict: ...
```
Da `model_dump()` in allen Beispielen identisch mit `to_context()` ist, könnte die Default-Implementierung `to_context = model_dump` sein. Kein BLOCK — die aktuelle Lösung ist funktional korrekt.

---

## [SUGGEST] S3 — `OrchestrationError` Exception-Hierarchie

**Datei:** ADR-096:239-243
**Begründung:** `OrchestrationError` wrapping ist korrekt, aber für Consumer-Apps wäre eine differenzierte Hierarchie hilfreich:
- `OrchestrationError` (base)
  - `ConfigResolutionError` (kein AIActionType gefunden)
  - `TemplateRenderError` (promptfw-Fehler)
  - `LLMExecutionError` (aifw-Fehler)

Consumer-Apps können dann gezielt auf Konfigurationsfehler vs. LLM-Fehler reagieren. Kann in der Implementierung entschieden werden.

---

## [QUESTION] Q1 — `_orchestrator = ChapterOrchestrator()` Singleton-Safety

**Datei:** ADR-096:416
**Kontext:** Der Consumer-App-Pattern zeigt einen Module-Level Singleton. `BaseContentOrchestrator` hat keinen State, daher ist das Thread-safe. Aber: Wird es auch mit Django's Auto-Reload (dev server) korrekt funktionieren? Falls ja, sollte das kurz in §4.6 dokumentiert werden ("stateless, thread-safe, survives auto-reload").

---

## [NITS]

- **N1:** §4.3 Zeile 248 — `llm_result: Any` Typ-Annotation: Wenn `aifw` bereits ein `LLMResult`-Typ exportiert, wäre eine konkretere Annotation besser als `Any`.

**Gesamturteil:** ✅ **APPROVED**

ADR-096 ist architektonisch ausgereift:
- **Klare Domain-Grenzen** (Writing / Research / Analysis) mit geteilter Basis
- **Korrekte Package-Hierarchie** (promptfw → aifw → authoringfw → consumer)
- **Typed Contracts** mit Pydantic v2 (frozen models, validated)
- **Explizite Non-Scope** Definition verhindert Gateway-Anti-Pattern
- **Composability** durch explizite Task-Pipelines (Research → Writing)
- **Test-Requirements** per ADR-057 definiert
- **Migration-Path** phasenweise und nicht-brechend

**Empfehlung:** Status auf `Accepted` setzen. Phase 1 Implementierung kann beginnen sobald aifw 0.6.0 (ADR-097) deployed ist.
