## [BLOCK] B1 — pyproject.toml: Poetry statt hatchling + Tippfehler

**Datei:** `ADR-103:870` — `[tool.poetry.dependencies]`
**Problem:** Platform-Standard (ADR-022) ist `hatchling` als Build-Backend.
ADR-103 nutzt Poetry (`[tool.poetry.dependencies]`). Außerdem Tippfehler:
- Zeile 880: `djangorestt framework` (doppeltes t, Leerzeichen)
- Zeile 882: `djangorestt framework-simplejwt` (doppeltes t, Leerzeichen)
- Zeile 888-889: `git+https` Dependencies für `platform-context`, `django-logging`,
  `django-health` — ADR-111 sagt: diese sollen als Packages von PyPI/GitHub Packages kommen.

**Fix:**
```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
dependencies = [
    "djangorestframework>=3.15",
    "djangorestframework-simplejwt>=5.3",
]
```

## [SUGGEST] S1 — `@dataclass` für RawAusschreibung statt Pydantic

**Datei:** `ADR-103:749-757`
**Begründung:** `RawAusschreibung` nutzt `@dataclass` — Platform-Standard ist Pydantic v2
für alle Value Objects (ADR-107, ADR-112). Für einen PortalAdapter-Input wäre
`BaseModel` mit Validierung (z.B. URL-Format, Deadline in Zukunft) robuster.

---

## [SUGGEST] S2 — BillingHubClient: synchrones httpx in async-Context

**Datei:** `ADR-103:806` — `httpx.get(...)` (synchron)
**Begründung:** Portal-Adapter sind `async def fetch_new()` (Protocol), aber
`BillingHubClient` nutzt synchrones `httpx.get()`. Wenn der Client aus einem
async View oder Celery-Task aufgerufen wird, blockiert er den Event-Loop.
Empfehlung: `httpx.AsyncClient` oder zumindest Hinweis dass der Client nur
in sync-Kontexten verwendet werden soll.
