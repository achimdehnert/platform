## Original-Review (verbatim)

### Steelman
Die Entscheidung löst ein real gemessenes, wiederkehrendes Problem (8 von 64 Repos mit dauerhaftem Typechange-Dirt, plus 6 Repos mit untracked Noise) mit dem minimal-invasiven Mittel `git rm --cached` + `.gitignore` — ohne den Working-Tree anzufassen und ohne Inhaltsverlust. Die SSoT bleibt unangetastet in `platform`, der Zielzustand ist in `iil-adrfw` bereits real bewiesen statt nur theoretisch, und die Exemption für `platform`/`platform-pinned` beruht auf einem dokumentierten Realvorfall (37 Typechanges, eingefrorener Refresh-Hook) statt auf Spekulation. Die drei Guards verschieben „clean" von einer einmaligen Aufräumaktion zu einer strukturellen Eigenschaft des Sync-Mechanismus selbst.

### Befunde (Original-Schweregrade)
- **AD-1** (mittel, Rollout): 64 Repos in 3 Orgs via Einzel-PRs = 64 Reviews, real vermutlich durchgewunken → Gate eher symbolisch.
- **AD-2** (hoch, Ignore-Guard): `SKIP-REPO` skippt Repos ohne wirksamen Ignore komplett → dauerhaft keine Workflow-Updates ohne Alarm.
- **AD-3** (mittel, Rollout): wer/was führt die initiale Remediation aus? Manuell über 14 Repos fehleranfällig, kein Skript erwähnt.
- **AD-4** (mittel, Konsequenzen): „Session-Start" einziger Trigger, kein CI/Cron-Fallback → nie geöffnete Repos behalten veraltete Workflows.
- **AD-5** (niedrig): kein Wort zu repo-spezifischen Overrides.
- **M28-1** (mittel): kein Sunset-Datum/-Kriterium für `sync-workflows.sh` trotz CC-first-Ablösung → Doppelwartung.
- **M28-2** (mittel): keine Tests für die drei Guards.
- **M28-3** (mittel): kein Dry-Run-Modus, kein zentrales Logging/Audit-Trail.

### Out-of-the-Box (Original)
1. Workflow-Inhalte per `platform-context`-MCP live ausliefern (Fetch statt Filesystem) — eliminiert das git-Index/Symlink-Problem an der Wurzel; setzt MCP-Laufzeit-Konsum voraus.
2. Git-Hook (`post-checkout`/`post-merge`) statt „Session-Start" als Trigger — schließt AD-4; selbst wieder ein Verteilproblem.

### Empfehlung (Original)
**Überarbeiten** (nicht ablehnen) — git-Mechanik solide; wichtigste Lücke = fehlende Beobachtbarkeit (`SKIP-REPO` kann Repos unbemerkt abschneiden).

### Vorgeschlagene Änderungen (Original)
1. REC-1→AD-2: `SKIP-REPO` sichtbar machen (Summary/Warnung).
2. REC-2→AD-3: initiale Remediation skriptgesteuert + `git ls-files`→0 als Check.
3. REC-3→AD-4: periodischer CI-Fallback über alle Repos.
4. REC-4→M28-1: konkretes Ablösekriterium für `sync-workflows.sh`.
5. REC-5→M28-2/3: min. ein Testfall pro Guard + `--dry-run`-Modus.
