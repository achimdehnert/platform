## [SUGGEST] S1 — CHECK-Constraint als Django Meta.constraints

**Datei:** ADR-095:188
**Begründung:** Der `CHECK (priority IN ('fast', 'balanced', 'quality'))` ist als raw SQL definiert. Platform-Konvention ist Django-native Constraints:
```python
class Priority(models.TextChoices):
    FAST = "fast"
    BALANCED = "balanced"
    QUALITY = "quality"

class Meta:
    constraints = [
        models.CheckConstraint(
            condition=Q(priority__in=["fast", "balanced", "quality"]) | Q(priority__isnull=True),
            name="aiactiontype_priority_valid",
        ),
    ]
```
Die SQL-Darstellung im ADR ist verständlich, aber bei Implementierung sollte Django-native genutzt werden.

## [SUGGEST] S2 — `_lookup_cascade` Logging bei Cache-Miss

**Datei:** ADR-095:246-269
**Begründung:** Bei einem Cache-Miss + 4-Step-Cascade auf der DB sollte ein `logger.debug()` den gewählten Step loggen. Bei Produktionsproblemen (z.B. unerwarteter Catch-All-Hit) hilft das bei der Diagnose. Kein BLOCK — die Cascade-Logik ist korrekt, nur Observability-Verbesserung.

---

## [QUESTION] Q1 — `check_aifw_config` als Management-Command oder CI-Step?

**Datei:** ADR-095:502, 516
**Kontext:** `check_aifw_config` wird in §7.2 und §8 erwähnt aber nicht definiert ob es ein Django Management-Command (`python manage.py check_aifw_config`), ein pytest-Fixture, oder ein CI-Workflow-Step ist. Für die Implementierung in Phase 1 sollte das festgelegt sein.
