## [BLOCK] B1 — Packages jetzt auf PyPI, nicht GitHub Packages

**Datei:** `ADR-111:70-80` (Option B)
**Problem:** ADR wurde am 2026-03-08 geschrieben. Seitdem wurden die Package-Repos
(`aifw`, `authoringfw`, `promptfw`, `weltenfw`, `illustration-fw`, `testkit`, `researchfw`)
als **eigenständige Repos** unter `achimdehnert/` aufgesetzt und publizieren direkt auf **PyPI**
(nicht GitHub Packages). `iil-testkit==0.1.0` ist bereits auf PyPI publiziert (2026-03-06).

Die ADR-Entscheidung "GitHub Packages als privater PyPI" ist damit **überholt** für diese Packages.
Die `packages/`-Subdirectory-Strategie in `platform` ist nicht mehr der aktuelle Stand.

**Fix:** ADR-Status auf `superseded` setzen oder v1.1 mit Klarstellung:
- Eigenständige Package-Repos publizieren auf PyPI (öffentlich)
- GitHub Packages nur relevant falls Repos privat werden
- `platform/packages/` Subdirectory ist deprecated zugunsten eigenständiger Repos

## [SUGGEST] S1 — `pip.pkg.github.com` URL ist veraltet

**Datei:** `ADR-111:206`, `ADR-111:215`
**Begründung:** GitHub hat die Packages URL auf `ghcr.io` (Container) und
`npm.pkg.github.com` (npm) umgestellt. Für PyPI ist die korrekte URL:
`https://pypi.pkg.github.com/achimdehnert/simple/` — nicht `pip.pkg.github.com`.
Verifizieren ob `pip.pkg.github.com` überhaupt noch funktioniert.

---

## [SUGGEST] S2 — `twine upload` Repository-URL prüfen

**Datei:** `ADR-111:167-169`
**Begründung:** `https://upload.pkg.github.com/achimdehnert` — diese URL-Form ist
möglicherweise veraltet. GitHub Docs empfehlen `https://upload.pypi.pkg.github.com/`
für Python Packages. Sollte vor Implementierung gegen GitHub Docs verifiziert werden.

---

## [QUESTION] Q2 — Org Secret vs. Repository Secret

**Datei:** `ADR-111:63-64`
ADR nennt "Org Secret wenn eine GitHub Organization vorhanden ist". Die Repos
liegen unter `achimdehnert` (persönlicher Account, keine Org). Org Secrets sind
nicht verfügbar. Ist das korrekt reflektiert?

## Gesamturteil

⚠️ **APPROVED WITH COMMENTS**

Der ADR ist architektonisch solide, aber durch die Entwicklung seit 2026-03-08
(eigenständige Package-Repos auf PyPI) teilweise überholt. Der eine BLOCK (B1)
erfordert eine Status-Aktualisierung — entweder `superseded` oder v1.1 mit
Klarstellung des aktuellen Stands.
