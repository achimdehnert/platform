# Review: ADR-114 — Discord als IDE-ähnliches Kommunikations-Gateway

**Reviewer:** Cascade (IT-Architekt / Senior Developer / Security)
**Datum:** 2026-03-11

**Ziel-Vision:** Discord als **verlängertes Cascade IDE** — bidirektionaler Chat mit Cascade von überall

## 🔴 BLOCK — Muss vor v2.0 gefixt werden

### [BLOCK-1] MADR-Frontmatter fehlt

**Datei:** ADR-114, Zeile 1-8
**Problem:** Kein YAML-Frontmatter (ADR-022 Platform-Standard). Status/Datum/Decision-Makers fehlen im maschinenlesbaren Format.

### [BLOCK-2] Layer 3 (GitHub Issues) ist kein "Chat" — Latenz Minuten bis Stunden

**Problem:** Das Kern-Versprechen "Discord = verlängertes IDE" erfordert Echtzeit-Interaktion. Layer 3 (Discord → GitHub Issue → Cascade sieht Issue in Windsurf → Antwort → Discord) hat Latenz von Minuten bis Stunden. Cascade muss Windsurf aktiv haben **und** das Issue manuell bemerken.

**Impact:** Für echten "Chat mit Cascade" ist Layer 3 ungeeignet als Primärpfad. Es ist ein Escalation-Path für komplexe Tasks, nicht der Hauptkanal.

**Fix:** Layer 2 (LLM Gateway) muss der Primärpfad werden. Layer 3 bleibt für Tasks die vollen Codebase-Zugriff brauchen.

### [BLOCK-3] Kein MCP-Tool-Zugriff von Discord — IDE-Kernfähigkeit fehlt

**Problem:** Cascade in der IDE kann SSH, Docker, GitHub, DB, DNS, Deploy via MCP-Tools ausführen. Discord kann davon **nichts**. Ohne Tool-Zugriff ist Discord kein "IDE", sondern nur ein Chat.

**Fix:** Layer 2 LLM braucht **Function Calling** mit MCP-Tool-Proxy:
- `deploy_check health` → Discord zeigt Ergebnis
- `docker_manage compose_ps` → Container-Status in Discord
- `git_manage status` → Repo-Status in Discord
- `ssh_manage exec` → Befehle auf Server ausführen (mit Role-Guard!)

Das macht Discord zum echten Remote-IDE: Fragen stellen UND Aktionen ausführen.

## 🟡 SUGGEST — Empfehlung

### [SUGGEST-1] 5-Layer Architektur statt 3

Aktuelle 3-Layer-Architektur deckt den "IDE-Verlängerung" Use Case nicht ab. Vorschlag:

```
Layer 0: Cascade → Discord     (proaktive Notifications via discord_notify) ✅
Layer 1: Quick Actions          (Slash Commands ohne LLM) ✅
Layer 2: LLM Chat + Tools      (GPT-4o + Function Calling + MCP-Proxy) 🔴 KEY
Layer 3: Cascade Bridge         (GitHub Issues für komplexe Tasks) ✅
Layer 4: MCP-Proxy              (Direkte Tool-Ausführung: /run, /docker, /git) 🔜
```

**Layer 2 mit Function Calling ist der Schlüssel.** Das LLM entscheidet selbst ob es ein MCP-Tool braucht — genau wie Cascade in der IDE.

### [SUGGEST-2] Conversation History via Discord Threads

Für echten Chat braucht der Bot Thread-basierte History. Code dafür existiert teilweise:
- `cmd_chat()` hat `thread: bool = False` Parameter ✅
- Aber: keine History-Persistenz zwischen Nachrichten im Thread
- Fix: Thread-ID → pgvector, vorherige Messages als Kontext laden

### [SUGGEST-3] `/run` Meta-Command für MCP-Tools

Statt für jedes MCP-Tool einen eigenen Slash Command: ein `/run` Meta-Command mit Autocomplete:

```
/run deploy-check health risk-hub
/run docker ps risk-hub
/run git status bfagent
/run ssh exec "uptime"
```

Guards: Nur `platform-admin` darf `/run ssh`, `devops` darf `/run docker` + `/run deploy`.

### [SUGGEST-4] `/code` Command — Code aus Repos in Discord anzeigen

```
/code risk-hub apps/risk/services.py 1-50
→ Discord Embed mit Syntax-Highlighting (```python Block)
```

Nutzt GitHub API (bereits in Settings konfiguriert). Macht Discord zu einem echten Code-Viewer.

### [SUGGEST-5] Phase-Status im ADR inkonsistent

- `discord_notify` ist ✅ implementiert, steht aber im Review-Dokument noch als "Phase 3 (Zukunft)"
- Layer 2 `/chat` Handler existiert im Code, aber ADR sagt "🔜"
- Implementierungsstand muss in v2.0 aktualisiert werden

## 🔵 QUESTION — Klärungsbedarf

### [QUESTION-1] Soll Layer 2 LLM eigenständig Aktionen ausführen dürfen (Function Calling)?

Wenn ja: Das LLM in Discord kann MCP-Tools aufrufen — z.B. Deploy triggern, Container neustarten, DB-Backup erstellen. Das ist mächtig aber braucht:
- Approval-Flow für destruktive Aktionen (Discord Buttons: ✅ Approve / ❌ Cancel)
- Audit-Trail aller Tool-Aufrufe
- Token-Budget-Cap (MCP-Calls kosten zusätzliche LLM-Tokens)

### [QUESTION-2] Ist `llm_mcp` auf hetzner-prod deployed?

Der `/chat` Handler in `handlers.py` ruft `llm_mcp_url: http://llm_mcp:8001` auf. Settings konfiguriert. Aber läuft der Service?

### [QUESTION-3] DM-Support für mobilen Zugriff?

`guards.py` blockiert DMs explizit (Zeile 55-60). Für "verlängertes IDE" unterwegs wären DMs nützlich — mit separatem Auth-Mechanismus (z.B. Discord User-ID Allowlist, die bereits in Settings existiert: `allowed_user_ids`).
