## [BLOCK] B2 — Auth-Konzept für S2S-Calls undefiniert

**Datei:** `ADR-117:71-77` (Szenario B)
**Problem:** Szenario B nutzt ein "S2S-Service-Token" für Weltenhub-API-Calls ohne spezifiziertes Auth-Schema. Wer erstellt das Token? Wie wird es rotiert? Ist es ein API-Key, OAuth2 Client Credentials, oder HMAC (ADR-118)?

Ohne definiertes Auth-Schema kann jeder Service mit dem Token in die Weltenhub-DB schreiben — kein Audit-Trail welche App den Call gemacht hat.

**Fix:** Explizites Auth-Konzept definieren:
```python
# weltenfw/backends/weltenhub.py
backend = WeltenhubBackend(
    base_url=settings.WELTENHUB_API_URL,
    # Szenario A: User-Token (OAuth2 / Session)
    token=user.weltenhub_token,
    # Szenario B: Service-Token (HMAC per ADR-118)
    service_token=settings.WELTENHUB_SERVICE_TOKEN,
    source_app="bfagent",  # Audit-Trail
)
```

## [BLOCK] B3 — Kein Error-Contract für WeltenhubBackend

**Datei:** `ADR-117:92-104`
**Problem:** `backend.create_world()` gibt ein `result`-Objekt zurück mit `result.ok` und `result.id`, aber es gibt keinen definierten Error-Contract. Was passiert bei:
- 401 Unauthorized (Token abgelaufen)
- 409 Conflict (Welt mit gleichem Namen existiert)
- 503 Service Unavailable (Weltenhub down)
- Netzwerk-Timeout

Ohne Exception-Hierarchy oder definierte `WorldResult.error_code` kann der Caller (bfagent) nicht sinnvoll reagieren.

**Fix:** Error-Contract in weltenfw definieren:
```python
class WorldResult(BaseModel):
    ok: bool
    id: str | None = None
    error_code: str | None = None  # "auth_failed", "conflict", "unavailable"
    error_message: str | None = None
    backend: str = "weltenhub"
```

## [SUGGEST] S1 — `weltenhub_world_id` sollte UUIDField sein, nicht CharField

**Datei:** `ADR-117:45` — `World.weltenhub_world_id = UUID`
**Begründung:** ADR sagt "UUID aus Weltenhub", aber definiert nicht den Feldtyp. In Django sollte das ein `models.UUIDField(null=True, blank=True, db_index=True)` sein — nicht CharField, nicht BigAutoField-FK (Cross-DB-Referenz, kein FK möglich).

---

## [SUGGEST] S2 — Retry + Circuit Breaker für WeltenhubBackend

**Datei:** `ADR-117:170-173`
**Begründung:** ADR nennt "Celery-Task für nicht-blockierende Erstellung" als Mitigation für Latenz, aber kein konkreter Retry-Mechanismus. `httpx` mit `tenacity` Retry + exponential Backoff wäre Standard. Bei mehreren Consecutive Failures sollte ein Circuit Breaker greifen, der automatisch auf Szenario B (lokal-first) umschaltet.

---

## [SUGGEST] S3 — Management Command `upload_worlds_to_weltenhub` braucht Idempotenz

**Datei:** `ADR-117:151-154`
**Begründung:** Das Command iteriert alle Welten ohne `weltenhub_world_id` und erstellt sie in Weltenhub. Wenn das Command abgebrochen wird und teilweise durchlief, muss der zweite Run idempotent sein. Der ADR sollte explizit sagen: "Prüfe vor Create ob Welt mit gleichem Namen/Slug bereits in Weltenhub existiert (GET vor POST)."

## [QUESTION] Q1 — Tenant-Zuordnung in Szenario B

Wenn eine Welt via S2S-Token (ohne User) erstellt wird — welchem Tenant gehört sie in der Weltenhub-DB? ADR sagt `Tenant = System/Shared`. Aber weltenhub nutzt Multi-Tenancy (ADR-109) mit `tenant_id` auf allen User-Data-Modellen. Wie wird ein "System-Tenant" dargestellt? Gibt es einen speziellen Tenant-Eintrag dafür? Was passiert wenn zwei verschiedene bfagent-User denselben System-Tenant nutzen — sind deren Welten dann gegenseitig sichtbar?

---

## [QUESTION] Q2 — weltenfw v0.2.0 Breaking Change?

Das ADR erweitert weltenfw um ein `backends/` Package. Ist das ein Breaking Change für bestehende travel-beat/bfagent Imports? Wenn ja, sollte es v1.0.0 sein (Semver Major). Wenn nicht, muss klar dokumentiert sein dass `from weltenfw import WeltenClient` weiterhin funktioniert.

## Gesamturteil

❌ **CHANGES REQUESTED**

3 BLOCKs müssen gefixt werden:
- **B1:** MADR Frontmatter (trivial, 2 min)
- **B2:** Auth-Konzept für S2S-Calls definieren (architektonisch wichtig)
- **B3:** Error-Contract für WeltenhubBackend (Code-Qualität)

Die Architektur-Idee (Weltenhub-DB als SSoT, weltenfw als Schreibkanal) ist solide. Die fehlenden Contracts (Auth + Error) sind aber kritisch für eine produktionsreife Umsetzung.
