## 🔴 BLOCK — Muss vor Akzeptanz gefixt werden

### [BLOCK] Aktivierungs-Endpoint ohne Rate-Limiting / IP-Allowlist

- **Datei:** ADR-118:91-103
- **Problem:** Der `POST /api/internal/activate/` Endpoint wird nur durch `BILLING_INTERNAL_SECRET` Header geschützt. Ein geleakter Secret ermöglicht unbegrenzte Tenant-Erstellung. Es fehlt eine Defense-in-Depth-Strategie.
- **Referenz:** Security / ADR-081
- **Fix:** ADR muss festlegen: (1) IP-Allowlist auf `127.0.0.1` / Docker-internes Netzwerk, oder (2) Rate-Limiting (z.B. 10 req/min), oder (3) beides. Mindestens eine zweite Schutzschicht neben dem Secret.

### [BLOCK] `tenant_id` als UUID im Subscription-Model widerspricht BigAutoField-Konvention

- **Datei:** ADR-118:85
- **Problem:** `tenant_id # UUID, billing-hub generiert und übergibt` — billing-hub generiert UUIDs als Identifier. Das Model-Snippet zeigt `tenant_id` ohne Feld-Typ. Wenn dies als `UUIDField(primary_key=True)` in der Ziel-App landet, ist das ein Platform-Verstoß.
- **Referenz:** Platform Convention (BigAutoField) / ADR-022
- **Fix:** ADR klarstellen: `tenant_id = models.UUIDField(unique=True, default=uuid.uuid4)` — als **sekundärer Identifier** neben dem BigAutoField PK. Nicht als PK verwenden.

### [BLOCK] Deactivate-Endpoint undefiniert

- **Datei:** ADR-118:114
- **Problem:** `POST /api/internal/deactivate/` wird im Trial-Ablauf referenziert, aber nirgends spezifiziert (Payload, Verhalten, was passiert mit User-Daten).
- **Referenz:** ADR-Vollständigkeit
- **Fix:** Request/Response-Schema analog zum activate-Endpoint definieren. Mindestens: `{"tenant_id": "<uuid>", "reason": "trial_expired"}`. Was passiert mit den Daten? Soft-Delete? Read-only-Modus?

## 🟡 SUGGEST — Empfehlungen

### [SUGGEST] HMAC-Signatur statt Shared Secret

- **Datei:** ADR-118:95-96
- **Begründung:** Ein statischer `BILLING_INTERNAL_SECRET` Header ist anfällig für Replay-Attacks. Besser: HMAC-Signatur über den Request-Body mit Timestamp (wie Stripe Webhooks es selbst tun). Referenz-Pattern: `stripe.Webhook.construct_event()`.

### [SUGGEST] Idempotenz des activate-Endpoints explizit festlegen

- **Datei:** ADR-118:91-103
- **Begründung:** Was passiert bei doppeltem Aufruf mit gleicher `tenant_id`? Muss idempotent sein (kein zweiter Tenant). ADR sollte `409 Conflict` oder `200 OK (already exists)` als Verhalten festschreiben.

### [SUGGEST] Fallback bei billing-hub Ausfall

- **Datei:** ADR-118:132
- **Begründung:** ADR benennt das Risiko korrekt ("Ausfall blockiert Neuregistrierungen"), definiert aber keine Mitigation. Empfehlung: Retry-Queue in der Ziel-App oder asynchrone Aktivierung über Celery mit exponential backoff.

### [SUGGEST] Model-Felder vollständig typisieren

- **Datei:** ADR-118:72-89
- **Begründung:** Die Pseudo-Modelle zeigen nur Feldnamen ohne Django-Feldtypen. Für ein ADR dieses Impacts sollten die Felder vollständig sein (CharField, DecimalField, etc.), damit die Implementierung eindeutig ist.

### [SUGGEST] `BILLING_INTERNAL_SECRET` Rotation dokumentieren

- **Datei:** ADR-118:105
- **Begründung:** Wenn der Secret kompromittiert wird, müssen alle Apps gleichzeitig rotiert werden. ADR sollte den Rotations-Prozess skizzieren (z.B. dual-secret Support während Übergang).

## 🔵 QUESTION — Klärungsbedarf

### [QUESTION] Wie wird die E-Mail-Verifikation gehandhabt?

- **Kontext:** Im Flow registriert sich der User auf billing-hub mit E-Mail + Passwort. Wird die E-Mail verifiziert bevor der activate-Call an die Ziel-App geht? Sonst können mit Fake-E-Mails Tenants erstellt werden.

### [QUESTION] Wo lebt das Passwort des Users?

- **Kontext:** User gibt Passwort auf billing-hub ein, aber die Ziel-App hat eigene Auth-DB. Wird das Passwort im activate-Payload mitgegeben (Security-Problem!) oder generiert die Ziel-App ein eigenes Passwort und schickt eine "Set Password"-Mail?

### [QUESTION] `research-hub` als Produkt — ist das bestätigt?

- **Kontext:** Zeile 18 listet research-hub als Produkt (research.iil.pet). In früheren Diskussionen war research-hub noch in der Konzeptphase ohne festes Preismodell.
