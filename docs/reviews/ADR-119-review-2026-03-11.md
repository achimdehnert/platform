## 🔴 BLOCK

### [BLOCK] `object_id = PositiveIntegerField()` inkompatibel mit BigAutoField

- **Datei:** ADR-119:42
- **Problem:** `PositiveIntegerField` hat max 2.147.483.647 — `BigAutoField` kann bis 9.2×10¹⁸. Bei einem GenericForeignKey auf ein BigAutoField-Model wird der FK abgeschnitten wenn die ID > 2^31 ist. Platform-Konvention (ADR-022) ist BigAutoField als PK.
- **Referenz:** ADR-022 / Django GenericForeignKey Dokumentation
- **Fix:** `object_id = models.BigIntegerField()` oder `models.PositiveBigIntegerField()` (Django 4.1+).

### [BLOCK] `consumer_context = models.JSONField(default=dict)` ohne Schema-Validierung

- **Datei:** ADR-119:48
- **Problem:** `JSONField(default=dict)` akzeptiert beliebige Daten. Jeder Konsument schickt unterschiedliche Strukturen (`outline`, `tone`, `pov` für bfagent; andere Felder für travel-beat). Ohne Schema-Validierung entsteht schnell Daten-Chaos und LLM-Prompt-Fehler.
- **Referenz:** Platform-Konvention (pydantic v2 für API-Validation)
- **Fix:** Pydantic-Schema pro `consumer_app` definieren. Mindestens:
  ```python
  class BfagentContext(BaseModel):
      outline: str
      tone: str = "neutral"
      pov: str = "third_person_limited"
  
  CONSUMER_SCHEMAS = {"bfagent": BfagentContext, "travel_beat": TravelBeatContext}
  ```
  Validierung im Serializer vor DB-Speicherung.

### [BLOCK] API-Endpoints ohne Auth-Konzept

- **Datei:** ADR-119:64-91
- **Problem:** Die API-Endpoints (`POST /generate/`, `GET /{uuid}/`, `PATCH /regenerate/`) haben kein definiertes Auth-Konzept. Wer darf generieren? Nur der Tenant-Owner? Jeder authentifizierte User? Externe Services (bfagent → weltenhub)?
- **Referenz:** Security
- **Fix:** ADR muss festlegen:
  - **Intra-App** (weltenhub-intern): Session-Auth + Tenant-Isolation (TenantAwareModel prüft)
  - **Inter-App** (bfagent → weltenhub): HMAC oder API-Token (analog ADR-118)
  - Welcher Fall gilt hier?

## 🟡 SUGGEST

### [SUGGEST] `GenericForeignKey` → Konkrete FK-Felder evaluieren

- **Datei:** ADR-119:40-43
- **Begründung:** GenericForeignKey (ContentType + object_id) ist flexibel, aber: keine DB-Level-Constraints, keine `select_related()`, keine FK-Integrity. Wenn die Source-Typen begrenzt sind (Scene, Location, World, Character), sind 4 nullable ForeignKeys performanter und sicherer. Nachteil: Schema-Änderung bei neuem Typ.

### [SUGGEST] `uuid` Feld → sekundärer Identifier klarstellen

- **Datei:** ADR-119:38
- **Begründung:** Analog ADR-118 v1.2: `uuid = UUIDField(unique=True)` ist korrekt als sekundärer Identifier. Aber ein expliziter Hinweis fehlt, dass der PK `BigAutoField` bleibt. Für Konsistenz mit ADR-118 den gleichen Hinweis-Kasten ergänzen.

### [SUGGEST] Async-Generierung als Pflicht, nicht als Empfehlung

- **Datei:** ADR-119:123
- **Begründung:** "Celery-Task empfohlen für async" — LLM-Calls dauern 2-30 Sekunden. Bei synchronem Request-Handling blockiert Gunicorn-Worker. Das sollte **Pflicht** sein, nicht empfohlen. Pattern: POST → 202 Accepted + Task-ID, GET → Poll auf Status.

### [SUGGEST] Versionierung des `authored_text`

- **Datei:** ADR-119:51
- **Begründung:** Bei `/regenerate/` wird der alte Text überschrieben. Wenn der Autor manuell editiert hat und dann regeneriert, ist die manuelle Arbeit verloren. Empfehlung: `AuthoredContentVersion` Model oder zumindest `previous_text` Feld für Undo.

### [SUGGEST] Rate-Limiting auf `/generate/` und `/regenerate/`

- **Datei:** ADR-119:65, 88
- **Begründung:** LLM-Calls kosten Token. Ohne Rate-Limiting kann ein fehlerhafter Client hunderte Generierungen auslösen. Empfehlung: `django-ratelimit` (z.B. 10/min pro Tenant).

## 🔵 QUESTION

### [QUESTION] Lebt `AuthoredContent` in weltenhub oder in weltenfw?

- **Kontext:** ADR-117 definiert eine Shared World Layer (`weltenfw`). Wenn `AuthoredContent` in weltenhub lebt, können bfagent und travel-beat nicht direkt darauf zugreifen (nur via API). Wenn es in `weltenfw` lebt, können alle Apps es als Library nutzen. Welcher Ansatz ist gewollt?

### [QUESTION] Was passiert wenn das Quell-Objekt gelöscht wird?

- **Kontext:** `content_type + object_id` mit `on_delete=CASCADE` auf ContentType — aber der eigentliche Source (Scene, Location) hat keinen DB-Constraint via GenericFK. Wird der AuthoredContent verwaist?

### [QUESTION] Wie wird `consumer_ref_id` über App-Grenzen aufgelöst?

- **Kontext:** `consumer_ref_id = "chapter-123"` — weltenhub kennt die Chapter-Tabelle von bfagent nicht. Ist das nur ein opaker String für den Konsumenten, oder soll weltenhub damit etwas anfangen?
